# Phase 01.1: Flow Analytics — Top Talkers + Per-Host Charts - Context

**Gathered:** 2026-05-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Dar ao operador **visibilidade contínua de tráfego** independente de ataque ativo: uma lista dos 20 hosts mais "barulhentos" agora e um gráfico de 60s por host para drill-down. Tudo em RAM, sem novo storage, sem novas detecções, sem mitigação.

**O que está dentro do escopo:**
- Tabela "Top talkers" (20 linhas) atualizada em tempo real
- Gráfico stacked area por protocolo (UDP/ICMP/Outros) com toggle bps↔pps para qualquer host
- Nova rota `/traffic` no dashboard com drawer lateral para o gráfico
- Dois endpoints REST novos (top-20 e per-host)
- Strings i18n pt-BR + en-US para tudo que aparece

**O que NÃO está dentro do escopo (são fases futuras, não scope creep nesta):**
- Retenção histórica além de 60s
- Filtros por hostgroup, exporter ou CIDR
- Alertas baseados em top talkers
- Export CSV/JSON
- Mudanças em detecção ou mitigação

</domain>

<decisions>
## Implementation Decisions

### Retenção e fonte de dados
- **D-01:** Reutilizar `internal/aggregate.Store` como única fonte de dados. Nenhum storage novo, nenhuma migration, PERS-03 (RAM-only hot path) intacto.
- **D-02:** Janela de visão = 60s × 1s buckets que já existem no `WindowSize`. Charts mostram exatamente esses 60 pontos, nada mais. Operador que quiser "última hora" terá que esperar uma fase futura.

### Top talkers
- **D-03:** Ordenação fixa por **bps** descendente (não configurável nesta fase).
- **D-04:** Top **20** linhas, fixo. Suficiente para ver os heavy hitters sem virar inventory list.
- **D-05:** Cada linha mostra: IP, hostgroup (longest-prefix-match já existente em `internal/config`), bps total, pps total, e tag do **protocolo dominante** (UDP/ICMP/Outros — aquele com maior bps na janela).
- **D-06:** Escopo é **global por instância**. Sem dropdown de filtro. Multi-tenant continua resolvido por systemd instantiated units (`mitigador@isp` / `mitigador@cliente`).

### Gráfico por host
- **D-07:** Visualização = **stacked area** com três séries: UDP, ICMP, Outros. Empilhamento mostra vetor dominante de relance.
- **D-08:** **Toggle bps ↔ pps** acima do gráfico. Default = bps (mesmo critério do top talkers, mantém consistência mental).
- **D-09:** 60 pontos (um por segundo). Sem downsampling, sem zoom. Eixo X = relativo ("há 60s … agora").
- **D-10:** Tempo de exibição: enquanto a view estiver aberta. Polling pára quando o drawer fecha.

### Biblioteca de chart
- **D-11:** **uPlot** (`uplot` npm, ~40 KB gz) como única lib de chart introduzida nesta fase. Não usamos ECharts (overkill para 60 pontos) nem rolamos SVG na mão.
- **D-12:** Criar componente Vue compartilhado (`web/src/components/HostTrafficChart.vue` ou similar) que embrulha uPlot. Componente recebe `{ buckets, mode: 'bps'|'pps' }` e cuida do ciclo de vida (init, update, destroy). Reaproveitável caso fases futuras adicionem outros gráficos.

### UI placement
- **D-13:** Novo item de menu no `SidebarNav.vue`: **"Tráfego"** (pt-BR) / **"Traffic"** (en-US).
- **D-14:** Rota nova: `/traffic` → `TrafficView.vue` (segue padrão dos outros views já em `web/src/views/`).
- **D-15:** Tabela top-20 fica na view; **clicar em uma linha abre um drawer lateral (Naive UI `NDrawer`)** com o gráfico do host selecionado. Drawer mantém a tabela visível atrás para o operador comparar.
- **D-16:** Mobile (`< 640px`): tabela vira card stack vertical (mesmo padrão do `DashboardView`). Drawer pode virar fullscreen modal no mobile — decisão de detalhe fica com Claude.

### Real-time updates
- **D-17:** **REST polling a cada 1s** enquanto a view estiver visível. Sem mudanças no SSE broker — o broker continua sendo só para `attack.*` events.
- **D-18:** Endpoints novos sob `/api/traffic/*` (autenticados + CSRF, mesmo group do `r.Group` em `internal/api/server.go`):
  - `GET /api/traffic/top20` → array de 20 talkers
  - `GET /api/traffic/host/{ip}` → 60 buckets do host (404 se host desconhecido)
- **D-19:** Polling pausa quando a aba/janela perde foco (`document.visibilityState`). Sem polling em background — economiza CPU do servidor e bateria do operador no celular.

### i18n
- **D-20:** Todas as strings em `web/src/locales/pt-BR.json` e `web/src/locales/en-US.json`. Chaves agrupadas sob `traffic.*` (`traffic.titulo`, `traffic.col.ip`, `traffic.tooltip.bps`, etc.). Sem strings hardcoded no template.

### Claude's Discretion
- Comportamento exato do drawer no mobile (fullscreen vs bottom sheet)
- Loading skeleton enquanto o primeiro poll não chega
- Empty state ("nenhum host com tráfego nos últimos 60s")
- Tooltip do gráfico (hover mostra timestamp + valores por série)
- Cores das três séries (UDP/ICMP/Outros) — escolher do tema dark Naive UI por consistência
- Tratamento de erro de rede (mostrar último snapshot + indicador de stale, ou banner de erro)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `aggregate.Store.Snapshot(ip, now, window)` — entrega exatamente os 60 buckets necessários, já com breakdown por protocolo. Endpoint `/api/traffic/host/{ip}` é praticamente um wrapper JSON em volta disso.
- `aggregate.Store.ActiveHosts(now)` — devolve todos os IPs ativos na janela. Endpoint `/api/traffic/top20` itera, computa bps somando buckets, ordena, corta em 20.
- `chi` router + `r.Group` autenticado em `internal/api/server.go` — padrão pronto para os endpoints novos.
- `NDrawer` da Naive UI — componente nativo para o painel lateral; suporta mobile fullscreen via prop.
- `api()` em `web/src/api/client.ts` — cliente REST com CSRF/cookie session, já testado.
- `vue-i18n` + arquivos de locale — caminho pronto para strings pt-BR/en-US.

### Established Patterns
- **Single source of truth:** o aggregate é a fonte; não duplicamos counters em outro lugar.
- **API mount group:** todos os endpoints novos entram dentro do `r.Group` autenticado em `server.go` para herdar `requireAuth` + `csrfMiddleware`.
- **JSON shape:** seguir o estilo dos handlers existentes (`{ items: [...] }` para listas, payload flat para item único).
- **Vue view skeleton:** loading/error/empty states explícitos (`NSpin`, `NAlert`, `NEmpty`) como em `DashboardView.vue`.
- **Mobile responsiveness:** breakpoint `640px`; tabela desktop vira stack de cards.
- **i18n primeiro:** zero string hardcoded; todas via `t('...')`.
- **Polling com cleanup:** `setInterval` em `onMounted`, `clearInterval` em `onUnmounted`. Já tem precedente no `DashboardView.vue` (`durationTicker`).

### Integration Points
- **Wire-up server:** `cmd/mitigador/serve.go` constrói `api.Deps`; adicionar `*aggregate.Store` lá. `api.New(deps)` precisa receber o Store. Handler novo lê do Store via deps.
- **Router/Sidebar:** uma linha em `router/index.ts` + uma entrada em `SidebarNav.vue` + chaves novas nos JSONs de locale.
- **Bundle:** `uplot` entra como `dependencies` (não dev), `pnpm add uplot` (ou `npm install` segundo o que o projeto usa).

</code_context>

<specifics>
## Specific Ideas

- O insight do operador veio do smoke test da Phase 1: ele queria "ver o que tava rolando" quando nada atacava, e o dashboard só acende em ataque. Esta fase fecha esse gap específico.
- A inspiração mental é "htop para a rede do ISP": uma lista que muda em tempo real, ordenada por barulho, clicável para drill-down.
- Visualmente o stacked area por protocolo é o "tell" mais útil — UDP-amp fica óbvio, burst ICMP fica óbvio, sem precisar abrir uma tela secundária.

</specifics>

<deferred>
## Deferred Ideas

- **Retenção histórica > 60s** (roll-ups 5min/24h, ClickHouse ou Postgres) — esperar operador pedir explicitamente "quero ver ontem". Sem dados de uso ainda.
- **Filtros (hostgroup, exporter, CIDR)** — adicionar quando o volume de hosts ativos ficar alto o suficiente para o top-20 global não dar conta.
- **Configurabilidade do top-N** (10/20/50) e da ordenação (pps) — esperar feedback de uso real.
- **Alerting on top talkers** ("ping quando algum host passar X bps por T segundos sem ser ataque") — feature distinta, fase futura.
- **Export CSV/JSON** dos buckets — sem caso de uso ainda; pode entrar quando alguém pedir análise offline.
- **Mudar SSE broker para fan-out de counters** — só vale a pena se 1s de polling provar ser problema (overhead servidor ou bateria mobile). Hoje é especulação.
- **Side-by-side bps + pps top lists** — descartado nesta fase; toggle dentro do gráfico cobre o caso e mantém UI mais leve.

</deferred>
